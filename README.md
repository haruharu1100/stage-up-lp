# STAGE UP 法人営業LPプロジェクト 固定ルール

## 🎯 目的

法人営業リスト作成・企業別LP作成・DM送付準備を一貫して行う。

---

## ⚡ AI制作権限（絶対ルール）

LP制作時、AIは**最高品質と判断するデザイン・構成・コピーを自律的に選定する**。

- 配色・レイアウト・タイポグラフィ・余白・写真・アニメーション・セクション順・コピー文言：**すべてAI判断**
- 業種・社名・規模からAIが最適解を決定し、それを実装する
- 「どの色がいいですか」「どの構成にしますか」等の確認はせず、即座に最高案を実装する
- 承認待ちなしで commit → push → 公開まで完遂する
- 結果に対するレビューは可能だが、事前承認は不要

**これは絶対ルール。AIの裁量を信頼し、毎回最高品質を出す。**

## 🚫 L列・M列 絶対禁止ルール（最重要）

営業優先シート・タスク管理シート **どちらでも**：

- ❌ L列に「未送付」「DM済」「優先度」等の文字を入れない
- ❌ M列に「未実施」「面談済」等の文字を入れない
- ❌ ステータス・メモ・数式の自動入力をしない
- ❌ 既存値を上書きしない
- ✅ **L列・M列は完全に空のセルのまま**にする

→ 後で運用者が手入力するため、Claude/AI 側は**絶対に**何も書かない。

## 🏷 営業優先度の判定（内部判断用・スプレッドシートには記載しない）

⚠️ 2026-05-30 改訂以降、**スプレッドシートに「営業優先度」列は存在しない**。
ここに記載するのは**内部判断（LP作成可否・備考記載・対応順位の判断）**のための参考情報。
出力データには優先度の列を含めない。

### 高
- HPなし
- HPが古い
- スマホ未対応
- LINEなし
- 新設法人
- 建設業 / 工務店 / リフォーム業
- 集客改善余地が大きい

### 高（？）
- 業種不明
- HP不明
- 情報不足
- LP作成判断不可

### 中
- HPあり
- 最低限整備済み
- 改善余地あり

### 低
- 医療法人
- 宗教法人
- 公益法人
- 農事組合法人（半公的）
- 営業対象として不向き
- HP完成度が高い

## ❓ 業種不明ルール（改訂版 / 2026-05-30〜）

業種が判別できない場合でも **汎用コーポレートサイト型LPを作成する**。

```
業種予測：不明
LP URL：発行する（/_proposals/<slug>.html）
備考：業種不明のため汎用コーポレートサイト型LPを作成
※ 内部判断としては「高（？）」相当だが、スプレッドシートの列には記載しない
```

### 汎用LPの設計（テンプレ：`_template-generic.html`）
- 配色：**白ベース・ネイビー・グレー**
- フォント：Noto Serif JP（見出し）+ Noto Sans JP（本文）
- 構成：ファーストビュー / 企業理念 / 事業内容 / 選ばれる理由 / ご利用の流れ / FAQ / お問い合わせ
- タイトル例：「地域に根ざしたサービスで、お客様との信頼を築きます。」
- 事業内容は「地域に密着した事業」「お客様の課題解決をサポート」など汎用表現

### LP作成しないケース（営業対象外）
- 宗教法人
- 医療法人
- 公益法人
- NPO法人
- 半公的団体（協同組合・農事組合法人等）と明確に判断した場合

## 🏗 LP作成ルール（最重要）

### 大前提
LPは **STAGE UPの営業LPではない**。
その会社が実際に運営している公式ホームページとして作成する。

### ❌ 禁止事項
- 黒×金テンプレ流用
- 会社名だけ変更
- 料金プラン掲載
- SEO説明
- MEO説明
- LP制作説明
- STAGE UP説明
- 全社同じ構成
- 全社同じ配色
- 全社同じデザイン

### ✅ LP作成する業種
- 建設
- 工務店
- リフォーム
- IT / システム開発
- 士業（弁護士・税理士・司法書士・社労士・行政書士）
- 製造業
- サービス業（業態が明確なもの）
- 飲食
- 美容
- 教育（明確に業態がわかる場合のみ）
- 不動産
- 広告 / マーケティング
- ポスティング業

### ❌ LP作成しない
- 業種不明
- 情報不足
- 実態不明

業種不明は必ず「**高（？）**」で管理する。

### 🚫 DM文章の TSV 出力 絶対ルール（2026-05-30 制定・表示崩れ防止）

スプレッドシート貼り付け用 TSV を生成する際、O列（DM文章）は **必ず1セル1行** に収める。

- ❌ DM文章中に `\n`（バックスラッシュ + n）の 2 文字リテラルを残さない
- ❌ DM文章中に実改行コード（`\x0A` `\r`）を入れない
- ✅ 区切りは全角スラッシュ ` ／ ` を使用（例：`本文 ／ ■LP URL: ... ／ ■ID: sample ／ PW: sample1234 ／ ...`）
- ✅ 1セル = 1行 = タブで区切られる16列のうちの1つ

**理由**：セル内に `\n` リテラルや実改行が入ると、Google Sheets の折り返し設定によって行高が異常に広がり、シート全体の表示が崩れる。

### TSV 出力後の必須チェック（AI実施）

```bash
# 1. \n リテラル残存チェック（0件が正）
grep -c '\\n' _sheets/該当ファイル.txt

# 2. 列数チェック（データ行は必ず16列）
awk -F'\t' 'NR>=8 && NF!=16 {print "NG: "NR" 行が "NF" 列"}' _sheets/該当ファイル.txt

# 3. L/M列空白チェック（出力ゼロが正）
awk -F'\t' 'NR>=8 && ($12!="" || $13!="") {print "NG: "NR" 行 L="$12" M="$13}' _sheets/該当ファイル.txt
```

3つすべてパス（出力なし）で初めて完成。1つでも引っかかれば修正してから再チェック。

### 貼り付け先での表示設定（運用者実施 / `_sheets/_HOW_TO_PASTE.md` 参照）

| 項目 | 設定値 |
|---|---|
| **折り返し** | **「切り詰め」**（表示形式 → 折り返し → 切り詰め） |
| **行の高さ** | 21px（自動拡張禁止） |
| **O列（DM文章）列幅** | 320px 目安（広げすぎ厳禁） |
| **貼り付け方法** | `Cmd+Shift+V`（プレーンテキスト貼り付け） |

詳細手順とトラブル対処は `_sheets/_HOW_TO_PASTE.md` を参照。

## 🔄 法人営業リスト作成時のワークフロー（2026-05-30 改訂）

画像のみ渡された場合でも以下を実施：

1. 企業情報調査（社名・法人番号・住所・代表）
2. HP調査（WebSearch + WebFetch）
3. LINE調査
4. スマホ対応調査
5. **内部判断としての**優先度判定（スプレッドシート列には書かない）
6. LP作成可否判定
7. LP作成（業種判明 or 業種不明でも汎用LP）
8. GitHub反映（commit / push）
9. Vercel公開確認（curl）
10. LP URL記録（P列）
11. **営業優先シート** 用データ生成（A〜P列 / **L・M列 空白**）
12. **タスク管理シート** 用データ生成（LP URL あり社のみ / **L・M列 空白**）
13. L列・M列が完全空白か最終チェック
14. **O列 DM文章の改行/`\n`リテラル除去チェック**（grep / awk）
15. 16列構成チェック（awk NF）
16. 出力ファイル名は `_sheets/YYYY-MM-DD_<案件名>.txt` で固定保存

### タスク管理シート 反映条件（重要・2026-05-30 改訂）

タスク管理シートには、以下すべてを満たす会社**のみ**反映する：

1. 営業優先シートに登録済み
2. Q列（LP URL）が入力済み = **実際にLPが公開されている**
3. 営業対象外（NPO・宗教・医療・公益・半公的）でない

**反映しない会社：**
- LP URL が空欄
- LP 未作成（業種不明で保留）
- 営業対象外（10-farm / 21-mahoroba-coop / 25-mitsuhanoki / 29-munakata 等）

**今後の処理順序（固定）：**
1. 営業優先シートへ全社入力
2. LP作成対象のみLP制作
3. **LP URLが発行された会社のみ**、タスク管理シートへ反映
4. タスク管理シートの L列・M列は空白固定

## ⚡ スプレッドシート自動同期方式（2026-05-30 v5・運用フロー固定）

**Apps Script を毎回貼り替える必要は完全になくなりました。**

### アーキテクチャ

```
[Claude]                    [GitHub]                      [スプレッドシート]
   │                           │                                │
   ├─ build_data_json.py ─────▶ data/sales_list.json            │
   │                           │  (raw URL公開)                  │
   │                           │                                │
   │                           │ ◀── UrlFetchApp.fetch ─────────┤
   │                           │                                │ 「STAGE UP同期」
   └─ git push ────────────────▶                                │  メニュー実行
                                                                ▼
                                                          シート自動更新
```

### 構成ファイル

| ファイル | 役割 | 更新頻度 |
|---|---|---|
| `data/sales_list.json` | 全132社の正規データ（DM長文/短文両方を内包） | データ変更時に毎回 |
| `scripts/build_data_json.py` | JSON 生成スクリプト（業種判定・DM生成ロジック含む） | 業種テンプレ追加時のみ |
| `scripts/SyncSalesSheets.gs` | Apps Script 本体（JSON取得→シート反映） | **基本不変・初回1回のみ設置** |

### Claude 側の運用（毎回）

新規会社追加・既存社情報更新時：

```bash
# 1. LP作成 & auth.js/admin.html更新（既存ルール）
# 2. JSON 再生成
python3 scripts/build_data_json.py
# 3. push
git add data/sales_list.json scripts/build_data_json.py
git commit -m "data: 会社追加/更新"
git push origin main
```

### 運用者側の運用（毎回）

1. スプレッドシートを開く
2. メニュー「**STAGE UP同期 → 全シート一括反映（推奨）**」をクリック
3. 数秒で営業優先シート＋タスク管理シート両方が最新化される

**Apps Script の貼り替え・コピー・修正は不要。**

### 初回セットアップ（一度だけ）

1. スプレッドシート → 拡張機能 → Apps Script
2. 既存コードを全削除 → `scripts/SyncSalesSheets.gs` 全文を貼り付け → 保存
3. シートをリロード（F5）→ メニュー「STAGE UP同期」が現れる
4. 初回実行時のみ承認ダイアログで「許可」

### キャッシュ動作

- GitHub raw コンテンツの CDN キャッシュは最大5分
- Apps Script は `?t=<タイムスタンプ>` のキャッシュバスター付きで取得するため、push 後すぐ反映される
- 5分以上経過すれば必ず最新が取得される

### データソース確認方法

メニュー「STAGE UP同期 → データソース情報」をクリックすると、現在の JSON URL・生成日時・会社数が表示される（取得失敗時はエラー詳細）。
